✦ Onboarding Guide

Welcome to Arena AI

Arena AI is the AI workflow platform where teams build, deploy, and automate with AI agents. This guide walks you through every core concept — from the building blocks of your workspace to running, testing, deploying, and extending your workflows.

What is Mothership?

Mothership is the AI assistant in your main chat window — your always-on operator for the whole workspace. You talk to it in plain language and it builds workflows, manages tables and knowledge bases, runs research, schedules jobs, and takes actions across your connected integrations. Most of what you'll do in Arena AI starts as a conversation with Mothership. (Copilot, covered later, is the narrower assistant scoped to a single workflow you're editing.)

Your Workspace at a Glance

Everything in Arena AI lives inside your Workspace — a shared environment where your team's workflows, data, files, and knowledge are stored and managed. Think of it as your team's AI operating system.

The five core resource types you'll work with every day are:

📄

Files

Documents, presentations, CSVs, images — created or uploaded. Accessible to all workflows and Mothership.

📚

Knowledge Bases

Indexed collections of documents your AI agents can search and retrieve from at runtime.

💬

Tasks

Scheduled or immediate actions Mothership runs on your behalf — recurring jobs, alerts, direct actions.

📊

Tables

Structured workspace data — query, update, and export rows in plain language or from any workflow.

⚡

Workflows

Visual automation flows built from connected blocks. The core unit of automation in Arena AI.

Your First 15 Minutes

The hardest part of any new tool is the first session. You don't need to learn the canvas, blocks, or deployment to get value on day one — you just need to talk to Mothership about your own work. Here are four concrete things to type into the chat window in your own workspace.

1

Ask Mothership something

Open the chat window and type: "What can you help me with in this workspace?" Mothership knows your whole workspace and will orient you.

2

Create a knowledge base from your own docs

Type: "Create a knowledge base called My Docs and add the files I just uploaded." Attach 1–2 real documents. Now Mothership can answer questions from them.

3

Ask a question against it

Type: "Using My Docs, summarize our refund policy" (or any question your docs cover). This is retrieval working automatically — no setup.

4

Build and deploy one simple workflow

Type: "Build a simple workflow that takes a question and answers it using My Docs, then deploy it as a chat link I can share." You now have a shareable assistant.

The point

You never had to leave the chat window or learn the canvas. Start by talking to Mothership about your own work — the structure comes later, only when you need it.

Knowledge Bases

A Knowledge Base is an indexed collection of documents that your AI agents can search at runtime using semantic (meaning-based) search. Instead of hard-coding information into prompts, you store it in a Knowledge Base and your Agent blocks retrieve exactly what's relevant when they need it.

What They're For

  • Product documentation — so agents can answer customer questions accurately
  • Support content — FAQs, troubleshooting guides, onboarding materials
  • Competitive intelligence — battle cards, market research, positioning docs
  • Internal policies — HR handbooks, runbooks, compliance documents

How to Build One

1

Create the Knowledge Base

Tell Mothership: "Create a knowledge base called 'Product Docs'" — or create it directly from the sidebar.

2

Add Documents

Attach files, paste text, or point Mothership at a URL. It processes and indexes content automatically.

3

Attach to a Workflow

Open any Agent block in your workflow and select the knowledge base. The agent will search it at runtime.

4

Keep It Current with Connectors

Connect Notion, Google Drive, Slack, GitHub, Confluence, HubSpot, and 25+ more sources to sync automatically on a schedule.

How It Works

When an Agent block runs, it performs a semantic search across all attached knowledge bases and injects the most relevant content into its context window — no prompt engineering required.

Tasks (Automation & Scheduled Jobs)

Tasks are things Mothership does for you — either right now or on a recurring schedule. You don't need to build a full workflow for every one-off or repeating action. Just describe it and Mothership handles it.

Direct Actions (Right Now)

Mothership can take immediate actions using credentials connected to your workspace:

What You SayWhat Happens
"Send a Slack message to #eng that the deploy finished"Posts to Slack immediately
"Email the Q3 report to jane@example.com"Sends via connected Gmail
"Create a GitHub issue: auth tokens not rotating on logout"Opens the issue in the specified repo
"Add a contact to HubSpot: Acme Corp, ceo@acme.com"Creates the contact via HubSpot API

Scheduled Jobs (Recurring)

A scheduled job is a Mothership prompt that runs on a cron schedule. On each run, Mothership reads your current workspace and executes the task as if you'd just asked it.

  • "Every morning at 8am, check the leads table for new entries and post a summary to #sales in Slack"
  • "Every Monday at 9am, pull last week's workflow run counts and write a report to the workspace"
  • "On the first of every month, export the billing table to CSV and email it to finance@"
Managing Jobs

You can pause, reschedule, or delete jobs at any time: "Pause the morning summary job" or "Change the sync job to run every 3 hours instead of 6."

Tables

Tables are structured data stores built into your workspace. Use them to persist leads, workflow outputs, errors, enrichment results — anything that needs to be queried, updated, and shared across workflows.

Creating a Table

Describe the schema to Mothership and it creates the table instantly:

  • "Create a leads table with columns for name, email, company, status, and created date"
  • "Set up an errors table with: id (text), message (text), workflow (text), timestamp (date), resolved (boolean)"
  • "Create a table that matches the structure of this CSV [attach file]"

Querying in Plain Language

  • "How many rows in the leads table have status 'qualified'?"
  • "Show me all records from the past 7 days where score is above 0.8"
  • "What's the conversion rate from 'contacted' to 'meeting booked' this month?"

Using Tables in Workflows

Tables are available to any workflow via the Table block. Reference a table by name — no additional configuration needed. Workflows can read rows, write new records, and update existing ones programmatically.

Workflows

Workflows are the core automation unit in Arena AI. They are visual flows built from connected blocks — each block performs a specific task, and data flows between them automatically. A workflow can be as simple as "receive a webhook, call an AI model, send a Slack message" or as complex as a multi-branch, multi-agent pipeline with loops and condition routing.

How to Create One

Describe what the workflow should do to Mothership, or drag blocks onto the canvas manually in the workflow editor. Mothership builds the workflow and opens it in the resource panel in real time.

  • "Build a workflow that takes a URL, scrapes the page, summarizes it with Claude, and sends the summary to Slack"
  • "Create a lead enrichment workflow that takes an email, looks up the company, and writes the enriched record to the leads table"

Key Workflow Concepts

Test vs. Run

When you execute a workflow, there are two distinct modes. Understanding the difference is essential before you go to production.

🔎

Test Draft Mode

Running from the canvas editor. Uses your current draft state. Every change is live immediately. Use this while building to validate block outputs and debug logic before publishing.

▶

Run Deployed Mode

Triggered via API, schedule, webhook, or chat. Always uses the active deployment snapshot — the version you last published. Canvas changes have no effect until you deploy.

The Golden Rule

Test on the canvas. Deploy when it's ready. Any live trigger always runs the deployed snapshot, not your draft. Always click Deploy / Update after making changes.

Execution Flow

1

Trigger fires

Starts from its trigger block — Start, Webhook, Schedule, or Chat.

2

Dependency analysis

The engine maps block dependencies, identifies layers that can run in parallel.

3

Concurrent execution

Blocks with no mutual dependencies run simultaneously — no manual configuration needed.

4

Data flows forward

Each block's output is available downstream via {{blockName.output}} syntax.

5

Response returned

The first Response block that executes returns the result and ends the run.

Naming Your Workflows

When you say "Run the invoice workflow" or "Deploy the lead scorer as an API", Mothership matches on the name. Vague or duplicate names cause ambiguity. Clear, unique names make every Mothership conversation faster and more reliable.

Logging & Monitoring

Every workflow run is logged automatically — manual, API, Chat, Schedule, or Webhook. You have full visibility into what happened, what data flowed where, and exactly why something succeeded or failed.

💻

Real-Time Console

During manual runs in the editor, logs stream live on the right panel. See active block highlighting, outputs, timing, and error status for each step as it runs.

📋

Logs Page

All runs across all triggers are logged centrally. Filter by time, status, trigger type, folder, or workflow. Use Live mode to watch executions in real time.

What Each Log Captures

  • Block inputs and outputs — exact data that entered and exited every block
  • Run timeline — start/end timestamps, total duration, per-block run times
  • Error details — which block failed, full error message
  • Cost breakdown — token counts and model costs per block
  • Workflow snapshot — frozen copy of the workflow at time of run for debugging
Security Note

API keys and credentials are automatically redacted from log inputs. You can inspect block data safely without exposing sensitive values.

Controlling Token Spend

Every workflow run records a full cost breakdown — token counts and model cost per block — in its logs. That visibility is the foundation for keeping spend predictable.

Ways to Keep Costs Down

LeverHow it helps
Per-run cost logsOpen any run's log to see exactly which block and model drove the cost. Optimize the expensive steps first.
Right-size the modelUse a smaller/cheaper model for simple steps (classification, routing) and reserve premium models for the steps that truly need them.
Knowledge bases over giant promptsRetrieve only the relevant content at runtime instead of stuffing everything into the prompt — fewer tokens per call.
Cache and reuseUse sub-workflows and tables to avoid recomputing the same result repeatedly.
Note for the team

Usage tiers, spend caps, and billing thresholds are set at the account level — see your account settings or your Arena contact to configure limits.

Why Deploy?

When you build a workflow in the canvas editor, it exists as a draft — only you can run it manually from the editor. Deployment is the act of publishing it so it becomes accessible from the outside world: via an API endpoint, a shareable chat URL, a scheduled job, or a webhook.

Draft vs. Deployed

Think of the canvas as your scratch pad and a deployment as your production release. You can keep editing the draft freely — nothing changes for users or calling apps until you click Update to publish a new version.

The Three Deployment Types

🔗

API Deployment

Exposes your workflow as a REST endpoint. Any application can call it with an API key. Supports synchronous, streaming, and async execution modes.

💬

Chat Deployment

Creates a hosted conversational interface at a shareable URL. Supports multi-turn conversations, file uploads, and voice. Embed as an iframe anywhere.

🔧

MCP Tool

Exposes your workflow as a tool on an MCP server. External agents — Cursor, Claude Desktop, VS Code — can call it directly as a tool.

How to Deploy

1

Open your workflow

Navigate to the workflow in the canvas editor.

2

Click Deploy

The Deploy modal opens. The General tab shows current state and version history.

3

Choose a deployment type

Select API, Chat, or MCP from the tabs and configure the relevant settings.

4

Publish

Click Deploy (first time) or Update (subsequent releases). The workflow is now live.

5

Update after changes

When you modify the canvas, an "Update deployment" badge appears. Click Update to push the new version to production.

API Deployment

Once deployed as an API, your workflow is available at a stable REST endpoint:

POST https://agent.thearena.ai/api/workflows/{workflow-id}/execute

Authentication

By default, API endpoints require an API key in the x-api-key header. Generate keys in Settings → API Keys.

curl -X POST https://agent.thearena.ai/api/workflows/{workflow-id}/execute \
  -H "Content-Type: application/json" \
  -H "x-api-key: $ARENA_API_KEY" \
  -d '{ "input": "Summarize this article" }'

Execution Modes

ModeWhen to Use
SynchronousDefault. Wait for the complete response. Best for short-running workflows.
StreamingAdd "stream": true — receive token-by-token output as it's generated. Best for AI text responses.
AsynchronousReturns a run ID immediately. Poll for results separately. Best for long-running workflows.

Version Control & Rollbacks

Every Deploy or Update creates a new entry in the version history table. You can name versions, add descriptions, and roll back instantly.

ActionWhat It Does
RenameGive a human-readable label, e.g. "v2 — added error handling"
Add descriptionAttach notes about what changed in that deployment
Promote to liveMake an older version the active one — instant rollback, no canvas changes
Load into editorRestore a previous version onto the canvas for editing and redeployment
Rollback in Seconds

If a new deployment breaks something, find the last known-good version in the table and click Promote to live. All API, Chat, Schedule, and Webhook executions immediately switch to that version.

Blocks

Blocks are the building units of every workflow. Each block performs a specific, self-contained task. You connect them together and data flows automatically. Think of blocks as LEGO bricks — individually simple, combinable into any automation you can imagine.

Block Categories

CategoryBlock TypesWhat They Do
TriggersStart, Webhook, Schedule, ChatEvery workflow starts with a trigger. Defines how the workflow begins — manually, on a schedule, from an HTTP event, or a chat message.
AI & ProcessingAgent, Function, APIAgent blocks call AI models with tools and knowledge bases. Function blocks run custom JavaScript. API blocks call any external endpoint.
Control FlowRouter, Condition, Loop, ParallelDirect execution paths. Condition blocks branch on logic; Router blocks use AI; Loop blocks iterate; Parallel blocks run branches concurrently.
DataTable, Knowledge Base, MemoryRead and write workspace data. Semantic search from knowledge bases. Persist state across runs with Memory.
OutputResponseFormats and returns the final result. The first Response block to execute ends the run and sends the reply.
CompositionWorkflowCalls another workflow as a sub-flow. Always runs the deployed version of the child. Enables reusable, modular automation.

How Data Flows Between Blocks

Every block exposes its outputs as variables. Reference them downstream using double-curly syntax:

{{agentBlock.content}}       // Agent block text output
{{apiBlock.data.email}}      // Nested field from an API response
{{loopBlock.currentItem}}    // Current item in a loop iteration
Best Practice

Keep sub-workflows focused on one task. Small, single-purpose flows are easier to test, debug, and reuse across multiple parent workflows. Max nesting depth: 25 levels.

Integrations

Skills

Skills are reusable capability bundles you attach to Agent blocks. Instead of repeating the same instructions, tool configs, or knowledge base attachments across dozens of workflows, define a skill once and reference it everywhere it's needed.

What a Skill Contains

  • Instructions — persona, tone, domain expertise, behavioral rules for the agent
  • Tools — pre-configured tool access (web search, code execution, integrations)
  • Knowledge Bases — which knowledge bases the agent searches at runtime
  • Model settings — default model, temperature, and generation parameters
⚡

Reusability

Define once, use everywhere. Update a skill and every workflow using it immediately gets the improvement.

📝

Consistency

All customer-facing agents behave with the same tone and access the same knowledge — enforced at the skill level.

🏗

Modularity

Build skills for specific functions — "Support Agent", "Researcher", "Data Analyst" — and mix them across workflows as needed.

Example Skills

  • Support Agent — Product Docs knowledge base attached, instructed to be concise and never speculate
  • Researcher — web search and Firecrawl tools, instructed to always cite sources
  • Data Analyst — code execution and leads table access, returns structured JSON
  • Outbound Writer — persuasive B2B copywriter persona with access to competitor battle cards

Knowledge Base vs. Skills vs. Workflows — Do I Reference Them?

A common question: when you're in the Mothership chat, do you have to name your knowledge bases, skills, and workflows in your prompt, or does Arena AI use them automatically? Short answer: it's automatic. You can be explicit to disambiguate, but you rarely need to.

ResourceHow it's usedDo you reference it?
Knowledge BasesAgents search them automatically at runtime using semantic search.No — retrieval happens on its own. Naming one (e.g. "using Product Docs") only helps disambiguate when several could apply.
SkillsAttached to an Agent block, a skill is always active for that block. In Mothership chat, relevant skills are selected automatically based on your request.No — never invoke a skill by name. It's on when attached, or auto-selected in chat.
WorkflowsMothership matches workflows by name when you ask it to run or deploy one.Yes, loosely — say "run the lead scorer" and it matches by name. This is exactly why clear, unique names matter.
Rule of thumb

Knowledge and skills are pulled in for you automatically. Workflows are the one thing you reference by name — so name them well. When in doubt, just describe what you want; being explicit is optional, not required.

Choosing a Model (You Usually Don't Have To)

Arena AI supports many LLMs — OpenAI, Anthropic, Google, Groq, and more. Power users like fine-grained control, but most users never need to pick a model.

Every Agent block and skill ships with a sensible default model, so you can build, run, and deploy without ever choosing one. The model selector is there when you want it — to optimize cost, speed, or quality for a specific task — but it's an option, not a requirement. If you're not sure, leave the default.

For the average user

Ignore the model dropdown. The defaults are chosen to work well out of the box. Only revisit it if you have a specific reason — a task that needs the strongest reasoning, or one where speed and cost matter more than polish.

Copilot

Copilot is your in-canvas AI assistant, scoped to the workflow you're currently building. It reads the full structure of your workflow, makes changes directly, and saves checkpoints so you can revert at any time.

💬

Ask Questions

Explain what any block does, debug a failed run, or ask how to implement a particular pattern. Copilot has full context of the open workflow.

✏

Build & Edit

Switch to Agent mode to let Copilot propose and apply changes — add blocks, configure settings, wire variables — all with natural language.

📄

Checkpoints

Every change saves a checkpoint. Hover over any Copilot message and click the checkpoints icon to revert to a previous state instantly.

Copilot vs. Mothership

Copilot is scoped to one workflow — it reads and edits that workflow's blocks and connections. Mothership has access to your entire workspace and can build workflows, manage tables, run research, schedule jobs, and take actions across integrations.

Frequently Asked Questions

What is the difference between a Workflow and a Task?
A Workflow is a visual automation flow built from connected blocks with a persistent structure you can version and deploy. A Task (scheduled job) is a recurring Mothership instruction that runs on a cron schedule and can call workflows, take direct actions, or run research. Tasks are conversational; workflows are structural.
When should I use a Knowledge Base vs. a Table?
Use a Knowledge Base when you want agents to semantically search unstructured content — documents, PDFs, web pages. Use a Table when you need structured, queryable data — leads, metrics, records with defined columns. Tables are for storing workflow outputs; Knowledge Bases give agents access to reference material.
I changed my workflow — why isn't the API returning new results?
API, Chat, Schedule, and Webhook executions always run the active deployment snapshot. After making canvas changes, click Update in the Deploy modal to publish a new version. Until then, all external calls use the previously deployed version.
What is the difference between Test and Run?
Test (manual run from the editor) uses your current draft — canvas changes are reflected immediately. Run (triggered via API, chat, schedule, or webhook) always uses the deployed snapshot. This lets you safely edit and test without affecting production traffic.
How do I deploy a workflow as a chatbot?
Open your workflow, click Deploy, go to the Chat tab. Set a URL slug, title, output field, welcome message, and access control (Public, Password, Email, or SSO). Click Launch Chat. Your chatbot is live at a shareable URL and can be embedded anywhere via iframe.
Can I connect multiple accounts for the same service?
Yes. You can connect multiple accounts per service — for example, two separate Gmail accounts. Each block independently selects which account to use. Useful when different workflows need different identities or permissions.
What happens if a deployment breaks production?
Open the Versions table in the Deploy modal, find the last known-good version, and click Promote to live. All API, Chat, Schedule, and Webhook executions immediately switch to that version. Your draft canvas is not affected.
What is the difference between Copilot and Mothership?
Copilot is scoped to a single workflow — it reads and modifies that workflow's blocks and connections. Mothership has access to your entire workspace and can build multiple workflows, manage tables, run research, schedule jobs, connect integrations, and take immediate actions across services.
How do I keep a Knowledge Base up to date automatically?
Set up a Connector from the knowledge base settings. Arena AI ships with 30+ connectors — Notion, Google Drive, Slack, GitHub, Confluence, HubSpot, Salesforce, and more. New content is added, changed content is re-processed, and deleted content is removed on every sync run.
What are Skills and when should I use them?
Skills are reusable capability bundles — instructions, tools, knowledge bases, and model settings — attached to Agent blocks. Use them when a consistent agent persona or tool configuration should appear across multiple workflows. Define once, and all workflows using that skill get any update immediately.
